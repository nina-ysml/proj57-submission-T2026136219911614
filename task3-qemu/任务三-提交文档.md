# 任务三：StarryOS / QEMU 上 ACT 模型推理部署文档

## 一、完成概况

在 StarryOS（RISC-V 64，QEMU 虚拟机）上成功完成 ACT 模型的 CPU 推理。推理程序为纯 C 语言编写，使用 ONNX Runtime C API 加载模型并执行前向计算，输出预测的动作序列。

## 二、关键数据

| 项目 | 数值 |
|------|------|
| 可执行文件大小 | 13 KB |
| ONNX 模型大小 | 194 MB |
| 运行时依赖库大小 | 36 MB（libonnxruntime 19M + libstdc++ 17M + libatomic 140K + libgcc 584K） |
| QEMU 分配内存 | 1 GB |
| 推理时长（加载模型+推理） | 44.85 秒 |
| 输出 shape | [1, 8, 3]（batch × chunk_size × action_dim） |

## 五、部署流程（可复现）

### 1. 导出 ONNX 模型

```bash
cd proj57
python export_onnx.py    # PyTorch model.pt → model.onnx
```

### 2. 预处理输入数据

```bash
# 将 JPEG 图片预处理为 float32 raw tensor（224×224，ImageNet 归一化）
python preprocess.py frame_000227.jpg image.bin state.bin action_q01.bin action_q99.bin
```

### 3. 交叉编译推理程序

```bash
riscv64-linux-musl-gcc -O2 \
  -I<onnxruntime/include> \
  act_onnx_c.c \
  -L<runtime_libs> -lonnxruntime -Wl,-rpath,/lib -lm \
  -o act_onnx_c_riscv64
```

### 4. 编译 StarryOS

```bash
git clone https://github.com/Starry-OS/StarryOS.git
cd StarryOS
make ARCH=riscv64 rootfs
make ARCH=riscv64 build
```

### 5. 注入文件到 rootfs

使用 `debugfs` 将推理程序、模型、运行时库注入 StarryOS 根文件系统镜像：

```bash
debugfs -w rootfs-riscv64.img
  # 运行时库 → /lib/
  write libonnxruntime.so.1 /lib/libonnxruntime.so.1
  write libatomic.so.1 /lib/libatomic.so.1
  write libstdc++.so.6 /lib/libstdc++.so.6
  write libgcc_s.so.1 /lib/libgcc_s.so.1
  # 推理程序 + 模型 → /root/act/
  write act_onnx_c_riscv64 /root/act/act
  write model.onnx /root/act/act.onnx
  write image.bin /root/act/input_bins/image.bin
  write state.bin /root/act/input_bins/state.bin
  write action_q01.bin /root/act/input_bins/action_q01.bin
  write action_q99.bin /root/act/input_bins/action_q99.bin
  # libc.so 符号链接
  cd /lib
  ln libc.musl-riscv64.so.1 libc.so
```

### 6. 启动 QEMU 并运行推理

```bash
qemu-system-riscv64 \
  -machine virt -m 1G -nographic -cpu rv64 -smp 1 \
  -kernel starryos_riscv64-qemu-virt.bin \
  -device virtio-blk-pci,drive=disk0 \
  -drive id=disk0,format=raw,file=rootfs-riscv64.img

# 进入 StarryOS shell 后执行：
/root/act/act /root/act
```

## 六、设计说明

### 为什么不直接在 StarryOS 中解码 JPEG

StarryOS 作为轻量嵌入式操作系统，不提供 JPEG 解码库（如 libjpeg）。推理程序采用"宿主机预处理 → 原始 float32 数据"的策略，C 程序仅读取二进制文件，减少 StarryOS 侧依赖。

### 为什么用纯 C 而非 C++

C++ 的异常处理、RTTI、iostream 等机制在 StarryOS 上存在兼容性问题。ONNX Runtime 提供完整的 C API，纯 C 程序可直接完成模型加载和推理，生成的 ELF 更简单，StarryOS 加载器更容易兼容。

### 运行时库选择

ONNX Runtime 的 riscv64 musl 版本从社区预编译获取（来源：czwdysj/act-starryos-deploy），配合 GCC 运行时库（libstdc++、libgcc_s、libatomic），共 4 个动态库即可满足推理需求。
